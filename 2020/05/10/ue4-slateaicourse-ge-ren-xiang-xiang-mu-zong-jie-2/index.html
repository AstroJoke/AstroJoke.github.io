UE4-SlateAiCourse项目总结（角色和资源）

项目 UE4

这一期我打算不放过多代码了，就直接总结下写内容的关系。

前景提要和需要暂时创建的类

在已有的类的基础上，在Content添加Blueprint/Temp文件夹，用来放置暂时的蓝图（后期会删掉）然后在里面写个蓝图继承于PlayerCharacter，然后命名为PlayCharacterBP，然后我们修改PlayerCharacter的时候就可以动态观察修改的方法。

先考虑我们需要怎么做（角色）

接下来们需要在考虑我们要写的东西是什么（指进入游戏后）

唔首先既然是沙盒游戏，那么必然需要有个MC（Minecraft）那么就需要一个Actor，同时我们又需要看到我们的Actor，所以我们还需要一个Camera来随时跟着我们的MC，这里就设计到了两个Actor；

然后我们要移动，所以有了动作蓝图，然后使用动作蓝图完成移动。

接下来移动了就涉及到了视角变换（好吧就是一个矩阵）视角变换，然后和蓝图一起绑定后，需要使用某种方法将代码传递到蓝图上，这就是Montage（法语剪接），然后完成之前的动作蓝图就可以下一步了。

恩作为游戏，我们有了一个人，这个人可以到处跑，但是跑了有什么用？得打怪啊！（啪，你就刚会上下左右你就起飞了）不对不对，应该要检查我们周围有什么药啊素材啊，然后捡起来，捡起来之后先出现在快捷栏（毕竟DNF早期捡到的药都是出现在快捷栏的）（诶快捷栏需要做诶），恩涉及到素材我们又需要了解这个这个物品的属性，唔物品属性用啥来解决呢？emmm诶JSON鸭，反正一个文档很小，存点属性又不累。

最后就是把快捷栏和游戏角色的state连接起来，恩基本完美了。

于是我们要做的任务基本都订好了，接下来就是如何写的问题了

如何大概的完成这个项目

Some Preconcepts

  1. 先介绍一个概念Component，理解为技能啊熟练度啊之类就行了，算是一个Actor的附加属性。这里涉及到两个New Component的函数

    CreateDefaultSubobject<T>();//构造函数使用
NewObject<T>();//随时随地，但是记得Delete小心内存泄漏

  2. UE4的资源加载

    //UObject

LoadObject<T>();//随时随地
//蓝图

    同时还有以下的两个全局函数

    StaticLoadObject<T>();
StaticLoadClass<T>();
//分别是早期的LoadObject<T>();和LoadClass<T>();

添加玩家模型和相机

我们需要在PlayerController添加骨骼模型（默认是Mesh）

static ConstructorHelpers::FObjectFinder<USkeletalMesh> StaticMeshThird(TEXT("SkeletalMesh
'/Game/Res/PolygonAdventure/Mannequin/Player/SkMesh/Player.Player'
"));
USkeletalMesh* hhh = StaticMeshThird.Objetct;
GetMesh()->SetSkeletalMesh(StaticMeshThird.Object);

然后进入之前的PlayerCharacterBP的里面去绑定对应Player，然后在蓝图的预览。接下来在上面的代码下设置是否只有自己可看，是否接受Decals，其碰撞类型，是否能够发生碰撞等属性。然后运行就能去蓝图浏览details了

这只是第三人称的，你还需要自己写一个第一人称的，方法类似。然后使用UPROOERTY方法来声明camera，之后设置按照你自己的想法和素材决定。（camera有offset，摄像机手臂，是否旋转和距离等参数）

动作蓝图和玩家移动

在之前我们建立的基类SlAiPlayerAnim中使用UPROOERTY方法声明速度和旋转并在.h中设定初值。

接下来我们需要使用判断跳跃和移动，接下来保存运行并回到虚幻编辑器，然后就可以编辑我们的“小人”动作绑定了。（蓝图不咋会，所以这部分建议直接看视频）

然后蓝图设计完成之后，通过代码在playerCharacter.cpp中获得动作蓝图，就可以使用很少的代码实现很多东西了，然后需要实现前后左右等各种的移动方法实现，但是人物还有其他的动作没有写。所以下一节会告诉你怎么实现移动和pawn之间的交互，以及视角切换

视角切换

视角切换的过程需要主要你从第一人称到第三人称切换的逻辑

若切换第一人称，则自身不渲染，第三人称是要渲染的，同时camera也是不一样的，毕竟第三人称你可以理解为你随身绑着一个相机，而第一人称则是以你所在存在一个camera和你一起同运动，逻辑是不一样的，所以需要自行体会如何处理交换的逻辑以及交换前后的状态。

对于角色的其他动作，实际资源自带且已经做成了Montage。所以我们将其直接设置属于UpperBody类（上半身），然后绑定到角色的播放序列里面（用蓝图）然后使用layered blend per bone混合相应动作（当然之前还有触发在playerAnim中声明对应指针，然后写代码就行了）（说的轻巧，写起来挺麻烦的，建议直接看教程）

制作快捷栏UI

创建UI自然也需要一个Widget（命名SlAiMenuGameHUDWidget）用来显示UI，而另外一个Slat widget名为SlAiShortcutWidget来写逻辑。

在SlAiShortcutWidget.h中声明gamestyle，物品信息指针网格指针以及容器是否初始化的变量，以及初始化容器方法。

HUD的Widget可以和MenuHUDWdget类似，直接CV然后改改就行。另外需要在HUD内设置一个SlAiShortcutWidget的共享指针，然后同样设置dpi等参数，写一个获取屏幕尺寸的函数以及绑定到UIScaler的静态方法，然后和前面文件类似写这些方法；接下来在构造函数内使用SOverlay然后加插槽“画”。

画完之后将每一个boarder我们视作为一个容器，然后在SlAiShortcutWidget.cpp中实现：初始化这些容器（包括创建，画这个boarder）（后续还要绑定键位的）

记得每次加了新的要”画“的东西需要去GameWidgetStyle里面声明笔刷（数组，因为有多个），然后运行后设置。（如果发现没有出现笔刷，我们创建后再修改）

物品属性（JSON)

有了快捷栏，但是我们物品属性都没有，那么我们自然需要一些办法去获得这些资源的属性，显而易见需要使用JSON（也有其他的可以用，但是这个比较快）。此处内容和总结1里面JSON内容差不多。（在type头文件声明对应的枚举类型，然后引入json相关头文件，写解析函数即可）（注意如果你用的是UEnum直接和前面的CV改改接口参数就行了）。

导入了物品属性后，还需要写物品从快捷栏消失的逻辑判断函数。然后从datahandle里面写相关数据处理的函数。

**需要重看：注册快捷栏到playerstate

因为涉及到widget和数据的交互，我们最好不要使用直接动数据的方法，而是声明一个委托的方式。（具体方式我也不懂，这个地方我看的一脸懵逼）（需要多看几次

碰撞交互

物品碰撞可以在虚幻编辑器里面设置（Engine-Collison）设置preset，然后创建一个新的配置文件，在之前的handobject（是之前注册快捷栏之后设置的右手插槽来放工具的，这里略过了）设置一个自己的碰撞（因为自带碰撞会有bug导致某些移动无法达成）（每一个工具都是一个类哦）

这个碰撞包括了根组件，静态模型组件，碰撞组件以及Overlay交互监测关闭（否则自带监测就会出现之前的情况），最后通过委托绑定哦监测方法到检测体（然后每一个类都得写一个- -）

手持物体切换与动作通知

我们已经写了一个添加物品到手上的函数，我们希望在handobject.cpp内实现一个物体切换函数（传入物体的ID，然后替换物体），注意物体切换前后的逻辑。老师说有个工厂方法，具体实现看老师写好的源代码就行了。

只不过我们之前一直都没有设置对应物体的对应动作，接下来就是在Controller里面新建动作通知。在PlayerController添加下述代码（也可以在蓝图里面修改）

    //修改预动作
    void ChangePreUpperType(EUpperBody::Type RightType);
//cpp的实现根据UpperType来决定动画

//.cpp
void ASlAiPlayerController::ChangePreUpperType(EUpperBody::Type RightType = EUpperBody::None)
{
    //根据当前手持物品的类型来修改预动作
    switch (SPState->GetCurrentObjectType())
    {
    case EObjectType::Normal:
        LeftUpperType = EUpperBody::Punch;
        RightUpperType = RightType;
        break;
    case EObjectType::Food:
        LeftUpperType = EUpperBody::Punch;
        //如果右键状态是拾取那就给拾取,拾取优先级高
        RightUpperType = RightType == EUpperBody::None ? EUpperBody::Eat : RightType;
        break;
    case EObjectType::Tool:
        LeftUpperType = EUpperBody::Hit;
        RightUpperType = RightType;
        break;
    case EObjectType::Weapon:
        LeftUpperType = EUpperBody::Fight;
        RightUpperType = RightType;
        break;
    }
}

然后运行就可以浏览啦。

资源读取

资源读取还是使用JSON进行读取（需要提前把资源包里的JSON看一波）然后需要在datahandle的头文件中需要提前创建好对应的数据结构，包括枚举和结构体。然后在对应的cpp中完成其构造函数（根据所给的JSON来写对应构造函数）。

同时需要对资源碰撞进行预配置（在虚幻编辑器里面）然后创建新的配置文件，对其可见和相机的追踪类型设置为ignore（方便后面射线监测？）

然后在Resoure类里面写随机数来决定是哪种形态的（资源包的树，石头等都有好几种姿态），然后构造逻辑是什么，销毁啥的自己写。

拾取物体、准星和CPP操控材质

拾取物品需要使用射线监测来对其进行判断，因此需要建立一个新的Slate Widget来实现射线监测。

然后需要在playerstate里面绑定这个类到player里面。同时还需要声明委托来从datahandle到playerstate的数据传递。

接下来们就可以使用射线监测来识别物品，但是为了拾取物品，我们需要一个准心，准心的作用有两个，识别方向和采取资源时准心变化。

准心获得采取进读条的方法是老师用蓝图做的，这里需要自己去看下。

写完准星完成后，需要把这个类加到PointerWidget里面，同样的需要在代码里面实现什么时候出现在屏幕里，出现的高宽，出现框的大小等。这里涉及到“画”，故老生常谈画刷。画完后加载静态材质，然后再强制类型转换为动态材质，然后需要对其编辑。

这个时候在对应h文件设置想要写的类型的构造函数，tick函数（override）和更新函数。同时需要对playercontroller注册一个修改准星的委托，注册函数是之前的更新函数。

之后再GameHUD里绑定修改准星的委托（之前有快捷栏，注册射线信息的委托）

之后运行会发现，这个准星自动就开始跑了，咋办，这就说明射线监测还没写完，至少需要判断工作状态

射线监测当前的工作状态

在之前，在项目设置-引擎-碰撞新建一个射线类型，然后preset。

之后去playercontroller思考要写什么东西：射线监测结果，射线绘制，射线监测进行函数。然后实现三个函数（注意由于第一人称和第三人称，所以需要case选择）。

同时还可以写准星的动态放大缩小显示等相关的函数，按需加（cv）

工作状态下，我们需要写一个行为状态机（当然这里面有些新的就需要在sourceobject里面声明新的东西了）

void ASlAiPlayerController::StateMachine()
{
    普通模式
        准星显示未锁定
        更新准星
    如果检测到敌人
        准星锁定模式
    如果检测到资源
        如果左键没有按下,在资源模式下右键没有特殊意义
            准星锁定模式;
        如果左键已经按下
            获取实际伤害
            让资源受到伤害并且获取剩余血量百分比
            更新准星
    如果检测到可拾取物品,并且两者的距离小于300
        改变右键预状态为拾取
        修改准星锁定模式
        如果右键按下
            把物品捡起来
}

当然也需要对不同物体的范围有不同的变化，这里需要在playerstate实现，具体实现略

生成掉落物和动态创建材质

掉落物的初始化，生成，碰撞，和掉落物的ID（即掉落物是什么），笔刷以及笔刷对应的设置都是需要提前写的，还需要绑定材质。需要在代码和编辑器里面设置。同时还需要渲染新的贴图。还要写一个物品“爆”掉的物理模拟（或者说粒子？），这个模拟就是需要一些力，然后搞个随机力就可以模拟瞎爆炸的动画了。

还需要在ResourceObject里面遍历生成掉落物并随机赋予个数，然后生成（当然必须要世界存在才能生成）。

动态球形检测

有了掉落物，那么我们怎么检测什么时候可以拿呢？不同物品大小由不一样，所以需要一个动态球形检测（个人理解和包围盒差不多甚至更简单，包围盒毕竟是相交盒子再判断是不是和物体交，而这个只需要相交就可以判断可以拿了）。（当然你也可以搞个地上的掉落物旋转操作啥的，多实现一个就行了）

大概就是对于每个掉落物在FlobObject里面实现一个检测函数，其中检测半径（自己设定），检测到玩家，关闭关闭可被检测属性。若有背包空间，删掉对应掉落物，然后背包相关设置。若失败则不删（override Tick函数），一定时间后消失，同时需要将可被检测属性需要重新打开。注意手动注销定时器，内存泄漏不是好文明。
